# 智能算法学习助手设计与实现

## 一、项目名称

个性化算法学习助手

## 二、项目简介

本项目围绕我个人在算法学习中的真实需求，设计并实现一个个性化算法学习助手。它以我平时积累的算法模板为基础，通过人工分层分类、标签整理、源码高亮展示和网页交互界面，帮助我在做题时更快定位相关算法模板，并进一步理解模板所属专题、适用题型、识别信号和常见风险。

相比于普通的 AI 问答工具，本项目更强调"基于个人知识库的持续学习支持"。它不是简单地把题目交给大模型得到答案，而是围绕我自己的模板积累和学习方式，建立一个可持续扩展的学习系统。当前已经完成模板查询、AI模板讲解、题图分析、用户选择验证、学习诊断报告、模板入库与模板维护等完整能力链，形成了一个从"题目输入→分析推荐→用户尝试→AI验证反馈→学习诊断→迁移强化"的完整学习闭环。

## 三、问题分析

在算法学习中，我面临的典型问题并不是"完全不会写代码"，而是"面对题目时，难以快速判断它应该归入哪一类算法，也很难立刻从记忆中调出最合适的模板"。

具体来说，这个问题主要体现在以下几个方面：

1. 学过不少算法和数据结构，但知识点分散，真正做题时很难快速定位到对应模板。
2. 会背一部分模板代码，但对模板和题型之间的对应关系理解不够稳定。
3. 经常知道"这题好像见过"，但说不清是二分、图论、背包还是某种数据结构维护。
4. 模板放在本地文件夹里，虽然积累越来越多，但缺少统一、直观、可检索的查询方式。
5. 即使找到了模板，也不一定马上知道这个模板的适用条件、常见误区和改造方向。

因此，本项目首先要解决的不是"让 AI 直接替我做题"，而是"帮助我把个人模板库真正转化成一个可用、可查、可理解的学习工具"。

## 四、核心难点与痛点

围绕上述问题，本项目希望解决以下几个核心痛点：

### 1. 模板分散，查找低效

原始模板主要存放在本地文件夹中，虽然已有一定分类，但整体仍更像代码仓库，而不是学习工具。查找过程依赖手动翻目录，效率较低。

### 2. 模板与题型关系不清晰

仅仅看到一个模板文件名，并不能立刻知道它适用于什么题、为什么要使用它、有哪些常见误区。缺少这种解释性信息，会导致模板停留在"会背不会用"的层面。

### 3. 学习过程缺少统一入口

当前模板、知识点、代码和题目分析之间是割裂的，没有一个统一界面将它们组织起来。这样很难形成稳定的学习流程。

### 4. 难以进一步扩展成真正的学习助手

如果只是继续增加模板文件，模板库会越来越大，但使用体验不会自然提升。因此，必须先建立一套结构清晰、可持续维护的查询系统，才能在此基础上继续叠加推荐、反馈和 AI 能力。

## 五、核心功能与设计策略

### 1. 总体设计思路

本项目采用"先搭建个人模板知识库，再逐步扩展学习功能"的路线。第一阶段不追求复杂的智能问答，而是先把模板库转化为一个稳定、可演示、可维护的网页查询系统，用以支撑后续更高层次的学习辅助功能。

整体思路如下：

1. 将本地模板库作为系统的数据基础。
2. 通过脚本扫描模板文件，生成网页可用的数据。
3. 对模板进行人工分层分类，统一展示结构。
4. 为模板补充人工标签，例如难度、适用题型、识别信号、常见风险、关键词。
5. 在网页中提供分类浏览、名称过滤、详情查看和代码高亮等交互能力。
6. 在这一基础上，为后续"题目到模板推荐"和"AI 辅助维护"预留扩展空间。

### 2. 当前已实现的核心功能

#### 2.1 模板库分层分类

系统没有直接把原始文件夹原样搬到网页上，而是额外建立了一层"展示层分类"。例如：

1. 将二分、搜索归入基础算法。
2. 将树状数组、并查集、线段树归入数据结构。
3. 将最短路、最小生成树、拓扑排序等归入图论。
4. 将基础 DP、背包问题、进阶 DP 分开整理。

这样做的作用是让模板查询更符合学习者的理解方式，而不完全受原始目录结构影响。

#### 2.2 模板元信息展示

为了让模板不只是"代码文件"，系统为每个模板增加了说明信息，包括：

1. 人工分类路径
2. 原始文件位置
3. 分类说明
4. 学习难度
5. 关键词
6. 适用题型
7. 识别信号
8. 常见风险

这些信息可以帮助我在查看模板时，快速理解这个模板"适合什么""如何识别""容易错在哪"，使模板从代码片段转变为带解释的学习条目。

#### 2.3 网页查询系统

当前已经实现一个本地网页查询系统，主要功能包括：

1. 左侧按专题分层浏览模板。
2. 支持名称、目录名和标签过滤。
3. 支持按专题快速筛选。
4. 右侧显示模板详细说明和源码。
5. 支持源码复制、长代码折叠、自动换行和高亮显示。
6. 支持收藏模板和最近查看记录。
7. 支持在算法查询页直接删除当前选中的模板，用于维护纠错。

这使得模板库从"本地代码文件集合"转变为"带说明的可视化学习入口"，也为后续继续叠加推荐和反馈功能打下了基础。

#### 2.4 页面分工与 AI 介入方式

为了让项目结构更清楚，当前版本已经把不同能力拆分到三个页面和一个局部功能中，各自承担不同任务：

1. **算法查询页**：作为主入口，负责模板分类浏览、关键词过滤、标签查看、源码阅读，是"找模板、看模板、理解模板"的基础页面。
2. **当前模板 AI 讲解**：嵌入在算法查询页的详情区域中，用户只有在需要时才主动触发。系统会读取当前模板的标题、分类、标签与源码，自动识别算法主题，并生成面向学习者的讲解内容。这一功能解决的是"已经找到模板，但还想进一步理解它"的问题。
3. **题目 AI 分析页**：作为独立页面，负责上传或粘贴题目截图，让外部多模态模型先理解题意，再从本地模板库中给出候选模板。同时支持用户在分析前输入自己的模板猜测，由 AI 验证选择是否正确。这一功能解决的是"面对题目，不知道该先看哪个模板"的问题。
4. **模板入库页**：作为独立维护页面，负责粘贴代码或上传 cpp，先进行 AI 分类，再自动建议文件名和存放目录，并将新模板直接写入本地模板库。这一功能解决的是"模板越来越多后，如何持续维护和补充"的问题。

这样划分后，项目的学习流程就更清晰了：

1. 如果用户已经知道自己想看什么算法，可以直接进入算法查询页。
2. 如果用户已经打开某个模板但理解不够深入，可以触发当前模板 AI 讲解。
3. 如果用户连题目所属方向都还不确定，可以先进入题目 AI 分析页，让系统先给出候选模板。
4. 用户可以在分析前先猜测一个模板名，AI 会评估该选择是否正确并给出个性化反馈。
5. 系统会自动累计用户的判断记录，按算法专题统计正确率，并支持一键生成 AI 深度诊断报告。
6. 如果用户刚整理出一段新模板代码，或发现当前模板库需要修正，可以进入模板入库页补充模板，并在算法查询页做删除纠错。

这种设计避免了把所有功能混在同一个界面里，也让 AI 的角色更加明确：它不是取代模板库，而是在"题目理解""用户选择验证""模板解释""学习诊断"四个关键节点提供辅助。

#### 2.5 用户选择验证与学习诊断

题目 AI 分析页新增了两个紧密配合的能力：

1. **用户模板选择验证**：在调用 AI 分析前，用户可以先在"你的模板选择"输入框中填入自己猜测的模板名。AI 除了推荐候选模板外，还会额外评估用户的选择是否正确，并指出可能混淆的概念。系统会自动将每次判断结果（正确/错误、用户选择、AI 推荐、所属专题）记录到 localStorage 中。

2. **学习诊断面板**：基于累计的分析历史记录，自动统计总分析次数、选择正确率、掌握专题数和薄弱专题数。每个算法专题都有独立的正确率条形图和进度展示。系统根据正确率自动给出状态提示（掌握良好 / 部分专题需加强 / 建议系统性回顾）。

3. **AI 深度诊断报告**：点击"生成诊断报告"按钮，系统会将历史学习数据（总正确率、各专题表现、近期错误记录）发送给 AI，生成一份包含五个部分的结构化诊断报告：整体评估、掌握良好的专题、需要加强的专题（并分析可能原因：概念混淆/推理断层/迁移困难）、具体学习建议、学习趋势分析。

这三个功能共同构成了一个"尝试→验证→记录→诊断→改进"的学习反馈循环，使系统从工具化查询提升为具有学习闭环的个性化助手。

### 3. 技术实现思路

当前阶段的技术实现较为清晰，主要由三部分组成：

#### 3.1 模板数据生成

通过脚本扫描模板目录，将模板文件读取出来，并生成统一的网页数据文件。该过程负责：

1. 过滤无关文件。
2. 按规则识别模板所属专题。
3. 自动补充一部分人工标签信息。
4. 输出前端直接可用的数据结构。

#### 3.2 前端页面展示

前端采用原生 HTML、CSS、JavaScript 实现，页面展示仍以本地静态资源为主，但为了支持模板新增、删除和数据同步，当前版本已经配套一个本地 Node 服务。页面负责：

1. 展示分类总览。
2. 构建目录树。
3. 响应过滤条件。
4. 渲染模板详情信息。
5. 对源码进行语法高亮和交互增强。
6. 在需要时调用本地保存/删除接口，使网页内的模板维护动作能直接同步到本地文件。

#### 3.3 本地服务与维护接口

为了满足"网页里直接维护模板库"的需求，当前项目新增了本地 Node 服务 scripts/web-server.mjs。它除了提供网页访问，还负责：

1. 暴露 /api/save-template，用于将网页中的新模板直接写入模板库目录。
2. 暴露 /api/delete-template，用于删除当前模板，并同步清理对应 AI 元数据。
3. 暴露 /api/ai-chat，作为 AI 接口的代理层，避免前端直接暴露 API Key。
4. 在保存或删除后自动重跑模板数据生成脚本，保证查询页立即读取到最新结果。

#### 3.4 可扩展维护机制

当前系统已经具备继续扩展的基础。后续新增模板时，只需要：

1. 将新模板文件放入模板库。
2. 根据需要补充分层规则和标签规则。
3. 重新生成数据。
4. 刷新网页即可查看结果。

这使得系统能够随着模板库规模增长而持续维护，同时保持展示结构的一致性。

在当前的版本中，这一机制又进一步扩展为"规则生成 + AI sidecar 精修 + 网页内维护入口"的双层维护方式：

1. scripts/generate-template-data.mjs 仍负责扫描模板库并生成基础元数据。
2. 新增的 scripts/ai-maintain-template-metadata.mjs 可以调用外部大模型，对指定模板补充更细的分类、说明、关键词、适用题型、识别信号和风险提示。
3. AI 结果会写入 data/template-metadata.ai.json，而不是直接改源码。
4. 模板入库页在保存模板时，也可以直接触发一次 AI 元数据补全。
5. 生成脚本会自动合并这份 sidecar 文件，因此前端页面读取到的是"规则基础 + AI 精修"后的结果。

这样设计的好处是：既保留了本地规则的稳定性，也让 AI 可以承担"辅助维护模板元数据"的工作，而不需要直接替代查询系统本身。

## 七、已完成的学习闭环

当前版本已经实现了一个完整的"输入→分析→尝试→验证→诊断→建议"学习闭环：

1. **输入**：用户上传题目图片或粘贴截图。
2. **AI 分析**：多模态模型提取信号，从本地模板库推荐候选模板。
3. **用户尝试**：用户可先输入自己猜测的模板名。
4. **AI 验证反馈**：AI 判断用户选择是否正确，指出错误原因和可能混淆的概念。
5. **记录与统计**：系统自动记录每次判断结果，按专题统计正确率。
6. **学习诊断**：可视化展示各专题掌握情况，生成 AI 深度诊断报告（含整体评估、强项、弱项、建议、趋势）。

相比最初的"模板查询系统"，现在的系统已经从"找代码"升级为"理解→判断→验证→改进"的完整学习过程。

## 八、后续扩展方向

当前版本的后续改进方向包括：

### 1. 错误类型细分

在用户选择验证中进一步区分错误类型（计算错 / 理解错 / 方法选择错），使 AI 反馈更加精准。

### 2. 动态策略调整

根据用户正确率自动调节 AI 讲解深度。正确率高时给出简略提示，正确率低时进行详细拆解。

### 3. 迁移练习生成

在诊断报告后增加"生成针对性练习"功能，让 AI 根据薄弱专题自动生成变式题目。

### 4. AI 辅助维护

当前版本已具备 AI 辅助维护原型：既可以通过独立的 Node 脚本读取模板源码、调用外部模型生成元数据；也可以在模板入库页保存新模板时自动补充 AI 元数据。后续可进一步自动化批量维护。

这样的设计可以保证项目既保留本地查询系统的稳定性，也具备 AI 扩展空间，更符合课程中"AI 参与有策略、有角色"的要求。

## 九、自我评价、反思与对比

我认为这个项目目前最有价值的地方，在于它不是一个脱离个人学习实际的通用工具，而是围绕我自己的算法学习过程搭建出来的系统。

它的有效性主要体现在：

1. 能够把原本分散的模板整理成统一入口。
2. 能够帮助我从"找代码"过渡到"理解模板"。
3. 能够追踪学习进度，自动统计各专题掌握情况。
4. AI 验证机制让我在每次分析时都有"先判断再验证"的学习习惯。

但目前也有明显不足：

1. 标签生成仍以人工规则为主，维护成本还可以进一步降低。
2. AI 诊断报告的反馈粒度还可以更细，如区分"计算错/理解错/方法选择错"。
3. 系统还缺少"迁移练习生成"能力，即根据薄弱专题自动出变式题目。

从与传统学习方式的对比来看，以往我的学习过程更多依赖手动翻找模板、回忆算法名称和重复阅读代码文件，效率较低，也难以积累系统性的使用经验。当前这一版系统已经把"模板管理""模板理解""题目分析""用户验证""学习诊断"五个环节统一到了一个界面中，形成了"输入→分析→尝试→验证→诊断→改进"的完整学习闭环。这种"先让用户自己判断、再由 AI 验证反馈"的交互设计，明显优于传统学习方式中"做完题看答案"的单向反馈模式。

从 AI 的有效性和局限性来看，AI 在以下方面效果显著：
- **题目信号提取与模板匹配**：多模态模型对题型特征的识别准确率较高，能有效缩小候选模板范围
- **用户选择验证**：AI 能很好地识别用户是否"把 dp 当成了贪心"或"混淆了最短路和最小生成树"这类概念性错误
- **诊断报告生成**：AI 能基于历史数据给出结构化的学习建议，避免了人工分析的主观性

AI 的局限性主要体现在：
- 对代码细节的解释有时不够精确，仍需人工核对
- 诊断报告的质量依赖历史数据量，数据太少时分析深度有限
- 不适合直接替代做题过程，更适合承担"辅助判断、验证和解释"的角色

总体来说，这个项目已经完成了从"模板仓库"到"个性化学习助手"的转变，具备了完整的学习闭环和可运行的实际效果。

## 十一、项目运行说明

为了保证演示时可以快速启动，我为项目准备了统一的本地启动方式。实际运行时，优先使用一键启动脚本，其次再使用手动命令作为备用方案。

### 1. 运行环境

项目当前依赖非常少，只需要以下环境：

1. Node.js：用于执行模板数据生成脚本。
2. macOS 图形环境：用于直接打开浏览器预览页面。

### 2. 推荐启动方式

在项目根目录执行以下命令：

```bash
./启动网页.command
```

该脚本会自动完成以下工作：

1. 检查 Node.js 是否存在。
2. 运行模板数据生成脚本，刷新网页使用的数据文件。
3. 在本地 8123 端口启动最新的 Node 本地服务；如果端口上已存在旧实例，会自动重启为最新版本。
4. 自动打开浏览器，进入项目首页。

如果只想先检查启动流程而不真正打开服务，可以使用：

```bash
./启动网页.command --dry-run
```

如果默认端口被占用，也可以改用其他端口：

```bash
./启动网页.command --port 8124
```

### 3. 手动启动方式

如果不使用一键脚本，也可以手动执行以下两步：

```bash
node scripts/generate-template-data.mjs
node scripts/web-server.mjs --port 8123
```

启动后，在浏览器访问以下地址：

```text
http://localhost:8123/index.html
```

### 4. 页面使用方式

项目当前包含五个页面：

1. **算法查询页**：用于浏览模板分类、查看标签说明、阅读源码、触发当前模板 AI 讲解，并可直接删除选中模板。
2. **题目 AI 分析页**：用于上传或粘贴题目图片，调用外部模型并返回候选模板，支持用户选择验证和学习诊断。
3. **模板入库页**：用于粘贴代码或上传 cpp，执行 AI 分类、自动建议文件名和目录，并将模板直接写入本地模板库。
4. **项目介绍页**：演示辅助页，包含完整的演示流程指引和话术。
5. **成果展示页**：项目展示页，以展示型页面呈现项目全貌。

启动后默认会进入算法查询页。如果需要演示 AI 分析或模板维护能力，可以点击页面顶部的页面切换按钮进入对应页面。

### 5. AI 分析页演示步骤

当前题目 AI 分析页已经内置一组适合演示的默认配置：

1. 默认接口地址为千问兼容接口。
2. 默认模型为 qwen3-vl-flash。
3. 默认补充提示词已做精简，适合低成本演示。

演示时可按以下顺序操作：

1. 打开题目 AI 分析页。
2. 填入自己的 API Key。
3. 上传题目图片，或直接在页面中粘贴截图。
4. （可选）在"你的模板选择"中输入你认为适合的模板名。
5. 点击"调用模型分析"。
6. 查看题意概括、识别信号、候选模板与推荐理由。
7. 如果填写了模板选择，查看 AI 对你选择的验证结果。
8. 点击候选模板，联动查看本地模板详情与源码。
9. 滚动到学习诊断面板，查看专题掌握统计。
10. 点击"生成诊断报告"获取 AI 深度分析报告。

### 6. 演示说明与限制

当前版本属于本地演示方案，重点是验证完整学习闭环，因此有以下说明：

1. API Key 只在浏览器当前会话中使用，不写入项目文件。
2. AI 调用通过本地 Node 服务的 /api/ai-chat 代理，前端不直接暴露 API Key。
3. 如果后续做正式部署，可扩展为更完整的服务端方案。

### 7. 模板维护页演示步骤

当前版本还支持一条适合演示的模板维护链路，可按以下顺序操作：
