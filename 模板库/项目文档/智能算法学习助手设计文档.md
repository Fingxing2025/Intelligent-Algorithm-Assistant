# 智能算法学习助手设计与实现

---

## （1）针对什么问题

### 学习背景分析

我是一名算法竞赛学习者。在日常的算法学习过程中，我的学习方式有以下几个显著特点：

- **偏图像化理解**：面对一道新的算法题，我更习惯先看题目描述中的图结构、数据范围、输入输出示例，从整体上把握题意，而不是从抽象数学公式入手。
- **理解驱动**：我学算法的目标不是死记硬背模板代码，而是理解"为什么这种题型应该用这个算法""这个模板的核心逻辑是什么"。
- **模板依赖型**：在竞赛训练中，拥有自己的模板库是必需的。模板不只是"存代码"，它需要支持日常查询、考前复习，以及随着学习深入不断更新维护。

### 核心问题

在我长期积累算法模板的过程中，发现了一个矛盾：**模板积累得越多，真正做题时反而越难快速找到最合适的模板**。具体表现为：

1. **题型判断困难**：面对一道新题，难以快速判断它属于二分、图论、背包还是某种数据结构维护。经常停留在"这题好像见过"但说不清具体类别的阶段。
2. **模板检索低效**：模板分散在本地文件夹里（50+个 cpp 文件），虽然已有一定分类，但查找过程依然依赖手动翻目录，效率很低。
3. **模板理解断层**：即使找到了模板，光看文件名（如 `dijkstra.cpp`）无法立刻知道它适合什么题、有什么常见坑、和当前题目之间有什么联系。模板停留在"会背不会用"的层面。
4. **学习过程缺少反馈**：做完题只知道对错，无法系统性追踪自己在哪些算法类别上反复失误，缺少数据驱动的学习反馈。

因此，我需要一个**按照题意快速匹配模板、能解释模板原理、能追踪学习进度**的系统——不是让AI直接替我做题，而是把自己的模板库整理成可查询、可解释、可验证、可诊断的学习助手。

---

## （2）解决什么痛点

围绕上述问题，本项目希望解决以下五个核心痛点：

### 痛点一：模板分散，查找低效

原始模板存放在本地文件夹中，整体更像代码仓库而不是学习工具。查找依赖手动翻目录，效率低，也无法按"适用题型"或"识别信号"等维度进行检索。

### 痛点二：模板与题型关系不清晰

每个模板只是一个 `.cpp` 文件，缺少适用题型、识别信号、常见风险等解释性信息。导致"会背模板但不会用"的问题。

### 痛点三：题目到模板的匹配依赖直觉

面对题目时缺少系统性的匹配工具，靠个人经验和记忆猜测应该用哪个算法。一旦判断错误，后续所有推导都是无效的。

### 痛点四：学习过程缺少统一入口

模板查询、算法学习、题目分析、进度追踪之间是割裂的，没有一个统一界面将它们组织起来，难以形成稳定的学习流程。

### 痛点五：学习效果缺少量化反馈

做完题只知道对错，无法按算法专题统计掌握情况，不知道自己哪些专题强、哪些专题弱，也难以针对性地调整学习策略。

## （3）如何解决痛点：核心功能 & 设计思路

### 3.1 总体设计思路

本项目采用"以个人模板库为核心，AI在关键节点辅助"的设计策略。核心思路是：

1. **将本地模板库作为系统的数据基础**：通过脚本扫描模板文件，生成结构化数据。
2. **建立展示层分类体系**：不为原始目录结构所限，按学习者的理解方式重新组织模板层级。
3. **为模板补充元信息标签**：每个模板附带难度、适用题型、识别信号、常见风险、关键词，使其从"代码片段"转变为"带解释的学习条目"。
4. **AI在四个节点精准介入**：题目理解时辅助模板推荐、用户判断后验证选择、找到模板后辅助讲解、积累数据后诊断薄弱环节。
5. **构建完整学习闭环**：输入→分析→尝试→验证→诊断→改进。

### 3.2 核心功能模块

系统包含**五大功能模块**，分布在三个主页面中：

#### 模块一：模板库查询系统（算法查询页）

- **分层分类浏览**：将模板按图论、动态规划、数据结构、基础算法等专题组织，支持按分类目录逐级展开
- **名称/标签检索**：输入模板名、目录名或标签关键字，实时过滤匹配结果
- **源码高亮展示**：完整的 C++ 语法高亮（关键字、类型、字符串、注释、数字），支持自动换行切换
- **模板元信息**：每个模板展示人工分类路径、学习难度、适用题型、识别信号、常见风险、关键词
- **最近查看**：自动记录浏览历史，方便快速回看
- **收藏模板**：收藏常用模板，从侧边栏快速访问
- **模板删除**：选中模板后可直接删除，同步清理 AI 元数据和空目录

#### 模块二：AI 模板讲解（算法查询页内嵌）

- **按需触发**：选中模板后用户主动点击"需要 AI 讲解"，而非默认自动调用
- **自动识别算法**：AI 读取模板标题、分类、标签和完整源码，自动识别算法主题
- **六维讲解**：从核心思路、适用场景、识别信号、常见误区、阅读建议、算法识别六个维度生成讲解内容
- **安全性**：API Key 仅存在当前标签页 session 中，不写入项目文件

#### 模块三：题目 AI 分析 + 用户选择验证（题目 AI 分析页）

- **图片/截图输入**：支持上传题目图片或直接粘贴剪贴板中的截图
- **多模态模型分析**：外部多模态模型先理解题意，提取关键算法信号
- **本地模板库推荐**：从用户的模板库中返回最匹配的候选模板（按置信度降序排列），而非直接生成题解
- **用户选择验证**（核心交互设计）：分析前用户可先输入自己猜测的模板名，AI 除了推荐候选外，还会判断用户选择是否正确，并指出可能混淆的概念。这是一种"先判断、再验证"的互动学习模式
- **自动记录**：每次判断结果（正确/错误、用户选择、AI推荐、所属专题）自动记录到 localStorage

#### 模块四：学习诊断与反馈（题目 AI 分析页内嵌）

- **统计仪表盘**：自动累计分析记录，实时展示总分析次数、选择正确率、掌握专题数、薄弱专题数
- **专题追踪**：每个算法专题独立统计，绿色进度条（≥70%）表示掌握良好，橙色（60-69%）表示一般，红色（<50%）表示需要加强
- **AI 深度诊断报告**：一键生成包含五个部分的结构化诊断报告——整体评估、掌握良好的专题、需要加强的专题（含错因分析：概念混淆/推理断层/迁移困难）、具体学习建议、学习趋势
- **历史记录管理**：默认折叠显示最近5条，支持展开全部，每条记录可单独删除

#### 模块五：网页内模板维护（模板入库页）

- **AI 智能分类**：粘贴 C++ 代码或上传 cpp 文件后，AI 判断该代码最接近模板库中哪个专题
- **自动建议**：AI 推荐文件名和存放目录，可手动修改
- **一键入库**：点击"加入模板库"直接写入本地模板库文件夹，查询列表同步刷新
- **删除纠错**：回到算法查询页选中误添加的模板，点击"删除当前模板"即可清理，并同步删除 AI 元数据和空目录

### 3.3 学习闭环设计

以上五个模块围绕一个完整的学习闭环运作：

> **输入**（上传题目图片） → **AI 分析**（提取信号、推荐候选模板） → **用户尝试**（先猜测模板名） → **AI 验证反馈**（判断正误、指出混淆概念） → **记录与统计**（按专题累计正确率） → **学习诊断**（生成 AI 诊断报告，分析薄弱原因并提供建议）

### 3.4 AI 角色设计

AI 在系统中扮演 **四个明确角色**，各有触发条件和职责边界：

| AI 角色 | 触发场景 | 职责 |
|---------|---------|------|
| **题目理解** | 用户上传题目图片并点击分析 | 多模态模型读取图片，提取信号，从模板库推荐候选（不直接生成题解） |
| **用户验证** | 用户在分析前填写了模板选择 | 判断用户选择是否正确，指出混淆概念，结合历史数据给出个性化建议 |
| **模板讲解** | 用户选中模板后点击"需要 AI 讲解" | 结合标题/分类/源码自动识别算法，从六个维度生成讲解（按需触发，非自动） |
| **学习诊断** | 用户点击"生成诊断报告" | 基于历史数据生成结构化诊断报告，分析薄弱原因并给出建议 |

### 3.5 技术实现

| 层 | 技术选型 |
|----|---------|
| **前端** | HTML5 + CSS3 + Vanilla JavaScript（无框架） |
| **后端** | Node.js (ES Modules)，本地 HTTP 服务器 |
| **AI 接口** | Node 原生 `fetch` 代理调用 OpenAI 兼容接口（默认千问 qwen3-vl-flash），超时 180 秒 |
| **数据** | 脚本扫描模板库生成 JSON 静态数据；AI 元数据通过 sidecar 文件维护 |
| **存储** | 前端 localStorage（模板浏览历史、收藏、分析历史记录）+ sessionStorage（API Key） |
| **维护** | `/api/save-template`、`/api/delete-template` 接口支持网页内直接操作本地文件 |

## （5）自我评价总结与展望

### 项目价值

我认为这个项目最有价值的地方，在于它**不是一个脱离个人学习实际的通用工具，而是围绕我自己的算法学习过程搭建出来的系统**。它的有效性体现在：

1. **把分散的模板整理成统一入口**：50+个模板从"本地文件"变成了"可检索的知识网络"。
2. **从"找代码"过渡到"理解模板"**：每个模板都带有适用题型、识别信号、常见风险等标签，不再是单纯的代码片段。
3. **追踪学习进度，量化掌握情况**：按专题统计正确率，让"我哪个专题薄弱"有了数据支撑。
4. **"先判断再验证"的互动模式**：AI 不是直接给答案，而是让用户先思考，再验证——这比被动接受答案的学习效果好得多。

### 与传统学习方式的对比

| 维度 | 传统方式 | 本系统 |
|------|---------|--------|
| 模板查找 | 手动翻文件夹，凭记忆 | 分类树 + 关键词搜索，秒定位 |
| 模板理解 | 文件名和代码分离，无标签 | 每个模板带8种元信息标签 + AI讲解 |
| 题目匹配 | 凭感觉选算法，命中率不稳定 | 上传题目图片，AI推荐候选 + 验证判断 |
| 学习反馈 | 做完题只知道对错 | 按专题统计正确率 + AI诊断报告 |
| 模板维护 | 手工分类存放 | 粘贴代码→AI分类→一键入库；查询页直接删除纠错 |

### AI 的有效性与局限性

**有效方面：**

- **题目信号提取与模板匹配**：多模态模型对题型特征的识别准确率较高，能有效缩小候选模板范围。
- **用户选择验证**：AI 能很好地识别"把 dp 当成了贪心"或"混淆了最短路和最小生成树"这类概念性错误。
- **诊断报告生成**：基于历史数据给出结构化的学习建议，避免了人工分析的主观性。

**局限性：**

- 对代码细节的解释有时不够精确，需人工核对。
- 诊断报告的质量依赖历史数据量，数据少时分析深度有限。
- AI 不适合直接替代做题过程，更适合承担"辅助判断、验证和解释"的角色。

### 项目不足与展望

当前版本的不足之处：

1. **标签生成仍以人工规则为主**，维护成本还可以进一步降低。
2. **AI 诊断报告的反馈粒度还可以更细**，如区分"计算错/理解错/方法选择错"。
3. **系统还缺少"迁移练习生成"能力**，即根据薄弱专题自动出变式题目。
4. **学习策略尚未动态调整**，如根据用户正确率自动调节 AI 讲解深度。
5. **个人学习数据可视化程度不够**：目前诊断面板以统计卡片+进度条为主，缺少更丰富的趋势图、对比图和知识图谱等可视化形式，难以直观展示长期学习轨迹和知识点关联关系。
6. **用户体验细节有待打磨**：如页面加载速度、移动端适配、交互引导、暗色模式切换等方面的体验还可以进一步优化，让使用过程更加流畅自然。

后续改进方向：

- 实现错误类型细分（计算错 / 理解错 / 方法选择错），使 AI 反馈更加精准。
- 增加"生成针对性练习"功能，让 AI 根据薄弱专题自动生成变式题目。
- 根据用户正确率自动调节 AI 讲解策略：正确率高时给出简略提示，正确率低时进行详细拆解。
- 进一步利用 AI 辅助元数据维护，降低人工整理标签的成本。
- **提升学习数据可视化**：引入正确率趋势折线图、各专题掌握雷达图、知识点关联图谱等可视化组件，让学习进度和薄弱环节一目了然，帮助用户更直观地感知自己的成长轨迹。
- **全面优化用户体验**：增加新手引导提示、快捷键支持、页面响应优化、移动端兼容适配、暗色模式等，使系统在视觉呈现和操作流畅度上持续提升，让个性化学习助手真正融入日常学习场景。

### 总结

本项目完成了从"模板仓库"到"个性化学习助手"的转变，实现了模板查询、AI讲解、题图分析、用户验证、学习诊断、模板维护六条核心能力链路，并构建了"输入→分析→尝试→验证→诊断→改进"的完整学习闭环。系统通过**一键脚本启动**，本地即可完整演示，能够稳定展示全部核心功能链路，满足课程作业对个性化学习助手的设计、实现与演示要求。

| `scripts/ai-maintain-template-metadata.mjs` | AI 辅助元数据维护 |
| `web/index.html` | 算法查询页主入口 |
| `web/ai-analysis.html` | 题目 AI 分析页（含学习诊断面板） |
| `web/template-maintenance.html` | 模板入库页 |
| `web/defense.html` | 答辩页（提词器 + 展示面） |
| `web/project-intro.html` | 演示辅助页 |
| `web/project-showcase.html` | 项目成果展示页 |
| `web/app.js` | 前端交互、AI 调用、学习诊断逻辑 |
| `web/styles.css` | 全局样式 |
| `web/data/template-data.js` | 自动生成的模板数据 |
| `data/template-metadata.ai.json` | AI 维护的模板元信息 |
| `启动网页.command` | macOS 一键启动脚本 |

### 运行方式

```bash
# 一键启动（推荐）
./启动网页.command

# 或手动启动
node scripts/generate-template-data.mjs
node scripts/web-server.mjs --port 8123

# 浏览器访问
# http://localhost:8123/index.html
# http://localhost:8123/defense.html  （答辩页）
```
