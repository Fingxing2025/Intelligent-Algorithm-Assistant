# 模板库维护说明

## 1. 当前维护链路

项目现在支持一条可落盘的 AI 辅助维护流程：

1. 模板源码仍然放在 模板库 目录下。
2. AI 维护脚本读取当前模板数据与源码，调用外部模型生成更细的元数据。
3. 生成结果写入 data/template-metadata.ai.json。
4. scripts/generate-template-data.mjs 会自动合并这份 sidecar 文件。
5. 重新生成后，网页查询页和题目 AI 分析页都会直接读取更新后的元数据。

这意味着 AI 的角色是“辅助维护模板元数据”，而不是直接替代模板查询系统。

## 2. 元数据文件格式

AI 维护结果保存在 data/template-metadata.ai.json，结构如下：

```json
{
	"version": 1,
	"updatedAt": "2026-05-28T00:00:00.000Z",
	"templates": {
		"er-fen-er-fen-cha-zhao-shi-yong-shi-li-cpp": {
			"relativePath": "模板库/二分/二分查找使用示例.cpp",
			"title": "二分查找使用示例",
			"source": "ai-maintained",
			"model": "qwen3-vl-flash",
			"updatedAt": "2026-05-28T00:00:00.000Z",
			"layers": ["基础算法", "二分"],
			"note": "一句话说明模板定位",
			"difficulty": "中级",
			"scenarios": ["适用题型 1", "适用题型 2"],
			"signals": ["识别信号 1", "识别信号 2"],
			"risks": ["风险 1", "风险 2"],
			"keywords": ["关键词 1", "关键词 2"]
		}
	}
}
```

支持覆盖的字段包括：layers、note、difficulty、scenarios、signals、risks、keywords。

## 3. 常用命令

先重新生成基础数据：

```bash
node scripts/generate-template-data.mjs
```

只查看本次将处理哪些模板：

```bash
node scripts/ai-maintain-template-metadata.mjs --dry-run --limit 3
```

只维护和图论相关的模板：

```bash
TEMPLATE_AI_API_KEY=你的密钥 node scripts/ai-maintain-template-metadata.mjs --match 图论 --limit 3
```

强制重跑已有条目：

```bash
TEMPLATE_AI_API_KEY=你的密钥 node scripts/ai-maintain-template-metadata.mjs --match dijkstra --force --limit 1
```

如果需要自定义接口和模型：

```bash
TEMPLATE_AI_API_KEY=你的密钥 \
node scripts/ai-maintain-template-metadata.mjs \
  --endpoint https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions \
  --model qwen3-vl-flash \
  --limit 2
```

## 4. 推荐维护流程

1. 先把新模板文件放入 模板库 对应目录。
2. 运行 node scripts/generate-template-data.mjs，确认基础分类已生成。
3. 运行 AI 维护脚本，为新增或重点模板补更细的说明和标签。
4. 再运行一次 node scripts/generate-template-data.mjs，确保网页数据同步到最新结果。
5. 启动网页后，在算法查询页检查 AI 维护后的说明是否合理。

## 5. 注意事项

1. 当前脚本默认只处理“还没有 AI 元数据”的模板；如需重跑，要加 --force。
2. AI 维护结果会覆盖生成脚本的部分推断字段，因此更适合用于精修重点模板。
3. 题目 AI 分析和当前模板 AI 讲解依旧是运行时能力；模板库维护是离线写回流程，两者职责不同。
4. 当前已验证本地 dry-run 和 sidecar 合并链路；真实模型写回仍需你在本地提供可用 API Key 后再执行。
